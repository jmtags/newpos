Case Management Module Flow

Operational guide for the Psyzygy Clinic POS Case Management module: how cases are created, tracked, assigned, updated, and reviewed.

Prepared for NEW_POS. Generated from the current implementation and database migrations.

1. Purpose

The Case Management module tracks the operational progress of clinic cases, especially assessment-related services that move through testing, scoring, interpretation, report writing, review, revision, release, and closure.

Important privacy boundary: the module is designed for operational progress tracking. Dashboards and case lists do not show full confidential clinical notes. The visible notes are progress remarks, task notes, and internal operational remarks.

2. Core Records

Record Purpose Important Links
cases Main case file used by the Case Management module. Client, service, transaction, transaction item, appointment, assigned associate.
case_progress_logs Status history and progress timeline for each case. Linked to one case. Stores from status, to status, notes, user/associate, and date.
case_tasks Operational tasks assigned to users or associates. Linked to one case. Tracks task status, due date, assigned user/associate, completion date.
services.requires_case_management Controls whether a POS service should automatically create a case. Configured in Service Settings per service.
mental_health_associates.user_id Links an associate login to the associate record. Used so associate users only see their assigned cases and tasks.

3. Case Creation Flow

A. Manual Case Creation

  1. A permitted user opens Case Management > Create Case.
  2. The user selects the client, service, assigned associate, priority, status, and target release date.
  3. The system creates a case record with a generated case number such as CASE-YYYYMMDD-XXXXXX.
  4. The case appears in All Cases, dashboard counts, and status-specific lists.

B. Automatic Case Creation From POS

  1. Admin or manager opens Service Settings and turns on Requires Case Management for a service.
  2. A POS transaction is created and includes that service as a transaction item.
  3. After the transaction item is inserted, the database trigger checks the service setting.
  4. If the service requires case management, the system creates one linked case.
  5. The case is linked to the client, service, transaction, transaction item, and associate if one was selected on the item.
  6. The default case status is New.
  7. The default report status is Not Started.
  8. For assessment services, the target release date defaults to transaction date plus 30 days.
Duplicate protection is built in. The system has a unique index on cases.transaction_item_id, so the same POS line item cannot create multiple cases.

4. Main Operational Workflow

A typical assessment case moves through this sequence:

  1. New - case has been created manually or from POS.
  2. Scheduled - testing or appointment schedule is set.
  3. Testing Ongoing - assessment/testing is in progress.
  4. Testing Completed - testing phase is done.
  5. Scoring - protocols/results are being scored.
  6. Interpretation - results are being interpreted.
  7. Report Writing - report is being drafted.
  8. For Review - report or case output is ready for review.
  9. For Revision - reviewer requested changes.
  10. Ready for Release - output is complete and ready to release.
  11. Released - output has been released to the client or authorized recipient.
  12. Closed - case is complete and no active work remains.
  13. Cancelled - case was cancelled.

Whenever the status changes through the Case Management API, the system automatically inserts a row into case_progress_logs. This gives the case details page a timeline of operational progress.

6. Case Tasks Flow

  1. A permitted user creates a task from Case Tasks or from a case detail workflow.
  2. The task can be assigned to a user or an associate.
  3. The task is given a status and optional due date.
  4. Task statuses are Pending, In Progress, Completed, and Cancelled.
  5. Associate users can update tasks assigned to them or tasks under their assigned cases.
  6. Completing a task records a completion timestamp.

The dashboard summarizes operational case workload using these cards:

Testing ongoingCases in Testing Ongoing status.
Report writingCases or reports in Report Writing.
For reviewCases or reports waiting for review.
For revisionCases or reports needing revision.
Ready for releaseCases or reports ready to release.
Released this monthCases released in the current month.
Overdue casesOpen cases past target release date.

Dashboard filters include date range, branch, associate, service, case status, report status, and payment status.

Branch filtering is prepared in the UI, but the current schema does not yet include a branch field. When branch data is added later, the dashboard can use it.

8. Role-Based Access

Role Case Access Allowed Actions Hidden/Restricted
admin All modules and all cases. Create cases, update status, assign associates, add progress notes, manage tasks, view payment status. None by design.
manager All modules and all cases. Same operational case permissions as admin, except admin-only system operations remain protected. None by design.
case_staff Case Management module. Branch scoping can apply when branch data exists. Create cases, update status, assign associates, add progress notes, manage tasks, view payment status. Transactions, payments, service pricing, clinic settings, discount types, audit logs, sales reports.
associate_user Only cases assigned to their linked mental health associate record. Update assigned tasks and add progress remarks for assigned cases. Payment details and non-case modules are hidden.
case_viewer Case progress viewing only. View case progress. No editing. Create, update, assignment, task editing, and payment details.
regular_user No Case Management access unless explicitly changed later. Existing regular POS permissions only. Case Management.

9. Payment Visibility

Payment status can appear on case rows/cards for admin, manager, and case_staff. It is hidden from associate_user because associate-level users should not see payment details.

The Case Management module uses transaction references for operational linking, but it does not expose full transaction/payment details to associate-level users.

11. Daily Recommended Workflow

  1. Configure services that require case tracking by enabling Requires Case Management.
  2. Create POS transactions as usual. Cases are created automatically for eligible service items.
  3. Open Case Dashboard to review new, overdue, for review, and ready-for-release cases.
  4. Assign associates and tasks where needed.
  5. Use status changes and progress remarks to keep the timeline updated.
  6. Use target release date and overdue list to monitor turnaround time.
  7. Release and close cases when operational work is complete.

12. Current Limitations and Notes

  • Branch filtering is prepared but branch data is not yet part of the current schema.
  • The module tracks operational progress, not full clinical documentation.
  • Automatic case creation happens only for services where requires_case_management is true.
  • Existing POS behavior remains unchanged for services that do not require case management.
